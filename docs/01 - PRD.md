# Product Requirement Document — FaCoderTerminal

## 1. خلاصه محصول

**FaCoderTerminal** یک برنامه ویندوزی با هسته Python است که به کاربر اجازه می‌دهد درخواست‌های فنی خود را به زبان فارسی بنویسد و برنامه آن‌ها را به دستورهای داخلی، امن و قابل‌اجرا تبدیل کند. برنامه به PowerShell / Windows Terminal / CMD / WSL متصل می‌شود و از یک LLM آنلاین کم‌هزینه برای تشخیص نیت کاربر استفاده می‌کند.

نکته کلیدی محصول این است که LLM اجازه ندارد آزادانه دستور سیستمی بسازد و اجرا کند. LLM فقط باید درخواست کاربر را به یک `command_id` شناخته‌شده در Knowledge Base تبدیل کند. سپس هسته برنامه، با استفاده از Command Catalog و Security Layer، تصمیم می‌گیرد چه چیزی اجرا شود.

## 2. مسئله‌ای که محصول حل می‌کند

کاربران فارسی‌زبان در کار با ابزارهای ترمینال و برنامه‌نویسی چند مشکل جدی دارند:

1. بیشتر دستورهای CLI، مستندات ابزارها و پیام‌های خطا انگلیسی‌اند.
2. ابزارهایی مثل Codex، Claude Code، Ruflo، GitHub CLI، Git، Docker، Laravel، Next.js و Python هرکدام دستورها و قواعد جدا دارند.
3. اجرای اشتباه یک دستور می‌تواند باعث حذف فایل، تغییر ناخواسته دیتابیس، آسیب به Git history یا توقف سرویس شود.
4. اتصال مستقیم LLM به Terminal بدون کنترل امنیتی قابل اعتماد نیست.
5. کاربر می‌خواهد فارسی بگوید «پروژه را بررسی کن»، «مشکل گیت را پیدا کن»، «با کد ایکس امنیت را چک کن» و خروجی دقیق، قابل کنترل و قابل بازبینی بگیرد.

FaCoderTerminal این مشکل را با یک لایه فارسی‌فهم، Knowledge Base دستورات، و اجرای کنترل‌شده حل می‌کند.

## 4. سناریوهای اصلی استفاده

### 4.1 اجرای دستور فارسی ساده

کاربر می‌نویسد:

> وضعیت گیت پروژه را بررسی کن

برنامه این درخواست را به `git.status` نگاشت می‌کند و دستور امن متناظر را اجرا می‌کند.

### 4.2 اجرای ابزارهای AI coding

کاربر می‌نویسد:

> با Claude Code پروژه را از نظر امنیتی بررسی کن، ولی فایل‌ها را تغییر نده

برنامه درخواست را به یک command template read-only تبدیل می‌کند و قبل از اجرا، جزئیات را به کاربر نشان می‌دهد.

### 4.3 اجرای دستور حساس با تأیید

کاربر می‌نویسد:

> مایگریشن دیتابیس را اجرا کن

برنامه تشخیص می‌دهد این عملیات حساس است، هشدار می‌دهد، و فقط بعد از تأیید صریح کاربر اجرا می‌کند.

### 4.4 تنظیم LLM کم‌هزینه

کاربر در Settings مقدارهای زیر را وارد می‌کند:

- Base URL
- API Key
- Model Name
- Provider Type
- Temperature
- Max Tokens

سپس برنامه از همان تنظیمات برای تشخیص نیت درخواست‌ها استفاده می‌کند.

### 4.5 توسعه Knowledge Base

برای هر ابزار، فایل Knowledge Base جدا ساخته می‌شود. نمونه ابزارهای هدف:

- Codex
- Claude Code
- Ruflo / MCP
- GitHub CLI
- Git
- Docker
- Laravel
- Next.js
- Python
- PowerShell

## 10. Acceptance Criteria

### 10.1 تنظیمات LLM

- [ ] کاربر بتواند Base URL را وارد کند.
- [ ] کاربر بتواند API Key را وارد کند.
- [ ] کاربر بتواند Model را انتخاب یا دستی وارد کند.
- [ ] دکمه Test Connection وجود داشته باشد.
- [ ] خطای اتصال واضح و فارسی نمایش داده شود.
- [ ] API Key در لاگ‌ها چاپ نشود.

### 10.2 تشخیص درخواست فارسی

- [ ] ورودی فارسی کاربر دریافت شود.
- [ ] خروجی LLM فقط در قالب JSON پذیرفته شود.
- [ ] اگر JSON نامعتبر بود، دستور اجرا نشود.
- [ ] اگر confidence پایین بود، برنامه از کاربر تأیید بگیرد.
- [ ] LLM فقط `command_id`های موجود در catalog را انتخاب کند.

### 10.3 اجرای دستور در PowerShell

- [ ] command template از catalog خوانده شود.
- [ ] دستور نهایی قبل از اجرا قابل نمایش باشد.
- [ ] خروجی stdout/stderr ذخیره شود.
- [ ] خطاهای اجرای shell در log ثبت شوند.
- [ ] برنامه دستور را در مسیر پروژه انتخاب‌شده اجرا کند.

### 10.4 Security Layer

- [ ] هر command_id دارای risk level باشد.
- [ ] عملیات high-risk بدون تأیید اجرا نشوند.
- [ ] عملیات blocklisted هرگز اجرا نشوند.
- [ ] policy بتواند الگوهای حذف، بازنویسی، reset، migration و تغییرات گسترده را کنترل کند.
- [ ] اگر دستور نهایی با policy تضاد داشت، اجرا متوقف شود.

### 10.5 Knowledge Base

- [ ] ابزارها در فایل‌های جدا تعریف شوند.
- [ ] هر command دارای id، title_fa، aliases_fa، command_template، risk و category باشد.
- [ ] loader بتواند KB را validate کند.
- [ ] command_id تکراری ممنوع باشد.
- [ ] command بدون risk معتبر نباشد.

## 13. Assumptions

- Assumption: نسخه اول فقط برای Windows 10/11 طراحی می‌شود.
- Assumption: زبان اصلی UI فارسی است؛ ترمینال و مسیرها LTR باقی می‌مانند.
- Assumption: LLM انتخابی از API سازگار با OpenAI یا مشابه آن پشتیبانی می‌کند.
- Assumption: برنامه در MVP اجازه اجرای command خارج از catalog را نمی‌دهد.
- Assumption: کاربر ابزارهای جانبی مثل GitHub CLI، Claude، Codex یا Ruflo را نصب می‌کند، اما برنامه وضعیت نصب را بررسی می‌کند.

## 14. تعریف موفقیت پروژه

MVP زمانی موفق است که:

- کاربر بتواند برنامه را روی ویندوز اجرا کند.
- LLM از تنظیمات کاربر متصل شود.
- حداقل 30 درخواست فارسی به command_id معتبر تبدیل شود.
- دستورات safe اجرا شوند.
- عملیات sensitive بدون تأیید اجرا نشوند.
- Knowledge Base قابل افزودن و validate باشد.
- خطاهای اصلی قابل ثبت و پیگیری باشند.
